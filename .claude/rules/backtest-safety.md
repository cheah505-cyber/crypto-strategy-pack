# 回测安全规则

涉及回测代码时自动启用。

## 杜绝未来函数

这是可验证的硬规则，不是建议。

### 核心约束

**每一个 `run_backtest()` 入口必须调用 `_preflight(df)`**，不通过则 `AssertionError`，回测不执行。`_preflight()` 检查：

- 所有信号列和指标列存在
- DatetimeIndex 严格单调递增、无重复（防 shuffle）
- 预热期后信号列零 NaN（shift/rolling 缺陷的直接证据）
- 信号列类型为布尔/整数（浮点信号可能携带未来信息泄露）
- OHLC 逻辑一致性

不调用 `_preflight()` 的回测脚本 = 不合规。

### 信号计算规范

信号计算函数（如 `compute_signals()`）中的每条赋值必须满足以下之一：

- 使用 `shift(N)` 确保引用历史数据（N ≥ 1）
- 使用 `.diff()` 或 `.pct_change()` 确保引用当前与前一 bar 的差
- 使用 `.rolling(W).mean()/.max()/.min()` + `shift(1)`（窗口统计含当前 bar 时，引用前必须 shift）
- 使用 `.ewm(alpha=..., min_periods=N)`（指数加权是因果的，但 `min_periods` 必须 ≥ 指标周期）

**禁止模式：**
- `df["close"].rolling(N).max()` 直接与当前 `close` 比较写信号（无 `shift(1)`）
- 缺失值用全局统计量填充（如 `fillna(df.mean())` — 全局均值含未来）
- 用当前 bar 的 `high`/`low` 决定当前 bar 的突破信号（必须先 `shift(1)`）

### 新增策略的验证门槛

新回测脚本上线前，以下必须全部满足：

1. `_preflight(df)` 嵌入 `run_backtest()` 入口
2. 跑一次全量回测，控制台出现 `Preflight PASS`
3. 检查控制台输出的交易次数与预期量级一致（数量级异常 = 信号逻辑错误）

违反任何一条 → 回测结果无效，不可作为决策依据。

## 数据分割

- 训练集、验证集、测试集严格按时间顺序分割，禁止随机分割
- 样本外区间至少占总数据时长的 20%
- 参数优化后必须在完全独立的样本外数据上验证

## 过拟合防护
- 参数优化后必须在完全独立的样本外数据上验证
- 记录每次回测的参数组合、随机种子、数据范围
- 多策略比较时使用相同的样本外区间
- 警惕"过度优化"——参数对小幅变化应相对稳定

## 费用和滑点

回测必须计入全部交易成本，严禁零成本假设：

- **手续费** — maker/taker 费率按交易所实际值设置，区分开仓和平仓
- **滑点** — 按该标的的流动性和时框如实估算，不得设为零或任意乐观值。大盘高流动性标的参考交易所盘口深度，低流动性标的留足安全边际
- **冲击成本** — 大仓位按成交量比例估算价格冲击，不得假设无限深度
- **资金费率** — 永续合约必须计入每 bar 资金费率
- **涨跌停/流动性限制** — 极端波动时可能无法以信号价格成交，需保守估算执行价

每次回测报告中显式声明使用的费用参数。费用参数变动视为策略变更，需重新跑全验证管线。

## 自动化优化循环

当跑参数扫描/自动调参时，以下规则防止自欺和过度优化：

### 一次一变
每次实验只改一个变量。同时动止损、仓位、因子权重 = 不知道谁起作用。结果不可归因。

### 评估器不可变
回测脚本是基准值。Agent 绝不能修改评估器——改了就等于换尺子，所有对比失效。硬门禁。

### 策略升级节奏

由浅入深，每阶段验证当前方向枯竭后再升级：

- **低垂果实** — 明显可改的小优化（参数微调、bug 修复、成本调整）
- **系统性探索** — 每个参数逐个遍历，网格搜索
- **结构性改动** — 算法替换、新因子引入、信号逻辑重写
- **激进实验** — 完全不同思路，推翻核心假设

阶段间切换条件：当前阶段连续多轮无显著提升时，才可升级到下一阶段。不得跳过阶段。

### 崩溃处理
- 笔误/缺 import → 修，重新跑
- 思路本身有问题 → 回退，记录 "crash"，跳过
- **5 次连续 crash → 暂停，告警。不继续烧算力**

### 每 10 轮复盘
分析 results 找规律（如"缓存类改动稳定提升 5-10%"，"重构类从未提升指标"），更新策略文档。后续轮次受益于积累的知识。

### 简洁优于复杂
小提升但引入丑陋复杂度 → 不值。同等性能但代码更简单 → 是赢。删掉代码但保持效果 → 最好结果。

### 超时
评估运行超过 2.5× 预期时间 → 杀掉视为 crash。可能是死循环或参数不可行。

### 蒙特卡洛验证
- 用 **bootstrap**（有放回重采样），不用 shuffle（打乱顺序结果不变——乘法交换律）
- 报告 P1/P5/P50/P95 收益分布，不只看均值

### Equity 计算检查

- **离谱数字先疑代码** — 收益或夏普远超该资产类别和时框的合理预期时，先假设有 bug，不要先假设找到了圣杯
- 常见陷阱：MTM 估值 + realized return 双重计算、全仓复利 on 部分仓位
- 检验方法：手动挑 3 笔交易，用计算器复算 equity 变化，确认与回测输出一致

### 回测 Sanity Test

每个 `run_backtest()` 调用前必须执行 `_run_sanity(df)`，验证引擎正确性：

- 强制单一入场持有至结束 → trade PnL = 价格变化减成本
- 全部信号关闭 → 零交易、零收益
- 固定时间点单笔交易 → PnL 与手动计算一致

不通过 → `main()` 返回非零退出码，回测结果不可信。

## 回测报告完整性
每个回测结果包含：
- 数据范围和时间粒度
- 总收益率、年化收益率、最大回撤
- 夏普比率、卡玛比率
- 交易次数和胜率
- 对比基准的表现

## 结果沉淀（闭环）

回测结果或策略参数发生变更后，必须同步更新：

| 目标 | 写入内容 | 类型 |
|---|---|---|
| auto-memory: `crypto-strategy-adx-perp.md` | 关键指标、验证状态、已知限制 | 原地覆盖 |
| `loop/findings.md` | 追加发现条目 | 只追加 |
| `lessons-crypto.md` | 可提炼的新教训 | 只追加（上限 20 条 → 归档见下文） |

不做此步骤 → 下一次会话基于过期数据决策。每次 git commit 回测结果时附带 memory 更新。

## Memory 归档

auto-memory 是工作缓冲区，不是永久存储。长期知识的终点是 Obsidian vault。

- **`lessons-crypto.md`**：上限 20 条。触及上限时，将已稳定（≥3 次被引用的）条目提升到 Obsidian `记忆/场景/` 或 `记忆/综合/`，从 memory 中移除。
- **`loop/findings.md`**：每个策略 / 每条验证链完成后，将完整发现记录归档到 Obsidian `记忆/场景/`（L2），清空 `findings.md` 只保留最近 50 行摘要。
